# polyfreq: Redesigning an R Package with Claude

*A case study in iterative performance engineering of a statistical genetics algorithm*

---

## 1. Project Overview and Goal

**polyfreq** is an R package implementing the de Silva et al. (2005) EM algorithm for estimating allele frequencies in autopolyploid populations under polysomic inheritance. The algorithm supports a mixed mating system — selfing at rate *s* plus random mating at rate *(1-s)* — and handles the "null allele" problem common in microsatellite genotyping.

The algorithm was originally available only through the `polysat` package, which called compiled C/C++ subroutines internally (in `deSilvaFuncs.cpp`). The goal of this project was to:

1. **Re-implement the full algorithm in pure R** — making the code readable, maintainable, and extensible without requiring a C compiler
2. **Scale to octoploid populations** (2n = 8x), where the full genotype space is astronomically large (e.g. C(27+8-1, 8) = 57,383,160 genotypes for a locus with 26 alleles)
3. **Match numerical results** of `polysat::deSilvaFreq()` to tolerance 1e-8
4. **Achieve runtime ≤ `polysat`** — and ideally much faster — via R vectorization, sparse matrix algebra, and C++ for the remaining bottlenecks

The driving dataset is a European hornbeam (*Carpinus betulus*) population genomics project with 8 microsatellite loci, 2 populations, ploidy = 8, and up to 34 alleles per locus.

## 4. Step 1 — Pure-R Implementation (Commit `826a025`, PR #1)

### What was built

All 9 functions were implemented from scratch in pure R, translating the C++ subroutines and the polysat R driver line by line:

**`R/helpers_desilva.R`:**
- `G_r`: replaced the C++ loop with a single `choose()` call — a vectorized C call, ~200× faster
- `INDEXG_r`: direct R translation of the combinatorial index formula
- `GENLIST_r`: iterative non-decreasing multiset enumeration (same algorithm as C++, pure R)
- `RANMUL_r`: vectorized via `t(apply(ag, 1, tabulate, nbins=na1))` + `lfactorial()` instead of per-genotype loops
- `SELFMAT_r`: the later bottleneck; translated the nested loop structure with vectorized gamete-pair batching

**`R/de_silva_freq_EM.R`:**
- `.make_indexf(m2)`: closure factory for the INDEXF phenotype-indexing function
- `.fenlist(na, m2)`: enumerates all observable phenotypes
- `.convmat(ng, nf, na1, ag, indexf)`: builds the phenotype-to-genotype conversion matrix
- `de_silva_freq_EM()`: the main EM loop — E-step (genotype frequencies via selfing equilibrium), M-step (allele frequency update), convergence check

### Key vectorization in the EM loop

The E-step originally used nested loops over genotypes and alleles. The vectorized version:
```r
pa_mat <- matrix(pa[ag], nrow = ng_int, ncol = m2)   # broadcast allele freqs over all genotypes
rvec   <- rmul * exp(rowSums(log(pa_mat)))             # multinomial random-mating freqs
gprob  <- (1 - self) * solve(diag(ng) - self * smatt, rvec)  # selfing equilibrium
```

The `pa[ag]` indexing trick — using the integer genotype matrix as an index into the allele frequency vector — eliminates all per-genotype loops.

### Challenge: numerical identity with polysat

Achieving bit-identical results required careful handling of:
- The `log(0) = -Inf` case: replaced with `-744` (minimum finite double exponent), matching polysat's C++ behaviour
- The convergence criterion: `sum(|p1 - p2| / |p1 + p2|) ≤ tol` with a guard for `|p1 + p2| < 1e-14`
- The selfing matrix divisor: `choose(m2, m)^2` applied to all entries, matching the C++ scaling

### Testing
- 106 pre-written tests covered all 9 functions
- Tests compare against `polysat::deSilvaFreq()` with tolerance 1e-8
- All 169 relevant tests passed on first implementation (4 pre-existing polysat failures excluded)

## 5. Step 2 — Sparse Path for Octoploid Scale (Commit `441c576`, PR #2)

### The problem: combinatorial explosion

For an octoploid locus with 17 alleles:
```
ng = C(17+1+8-1, 8) = C(25, 8) = 1,081,575 genotypes
```
The dense selfing matrix `smatt` would be 1,081,575 × 1,081,575 — around 4.4 TB of memory. The dense path is physically impossible for octoploid scale.

### The solution: union phenotype cone

The key insight from de Silva et al.: the EM algorithm only ever needs genotypes that are *compatible* with at least one observed phenotype. For a phenotype P = {a₁, …, aₖ} (the set of distinct alleles actually scored), the compatible genotypes are all m2-multisets drawn from {a₁, …, aₖ, null} — the "closure" of P.

The **union phenotype cone** is:
```
cone = ⋃_{P observed} { all m2-multisets from P ∪ {null} }
```

For the cb29/P8571 example: 818 unique phenotypes → 42,621 cone genotypes (vs 1,081,575 full space — **96% reduction**).

The cone is closed under selfing: any offspring of a cone genotype is also in the cone, so the EM within the cone gives identical results to the full-space computation.

### The Neumann series solver

The selfing equilibrium requires solving:
```
gprob = (1-s) · (I - s·S)^{-1} · rvec
```
where S is the (42,621 × 42,621) sparse selfing matrix. Instead of `solve()` (which would densify the matrix), the **Neumann series** converges geometrically:
```
(I - s·S)^{-1} · rvec = Σ_{k=0}^{∞} s^k · S^k · rvec
```
Each term is a sparse matrix-vector multiply. For s < 1 the series converges; termination when `s^k · max|v| < 1e-12`.

### Challenge: compact phenotype indexing

The INDEXF function maps a phenotype to an integer index in {1, …, nf} where `nf = Σ_{k=0}^{m2} C(na, k)`. For na=17, m2=8: nf ≈ 150,000. This was fine for indexing.

But for na=34 (the worst locus in the hornbeam dataset), nf would exceed 2.8 billion — impossible to allocate. Solution: **compact phenotype indices**. Instead of a full nf-length vector, track only the `n_uph ≤ 818` distinct phenotypes actually present in the cone, and use a compact 1..n_uph integer mapping.

## 6. Step 3 — Unordered Gamete Pairs (Commit `164f94e`)

### The bottleneck: selfing matrix construction

The sparse SELFMAT works by enumerating all (parent, offspring) pairs. An offspring genotype is formed by merging two gametes — each gamete is an m-subset of a parent's m2 alleles.

For octoploid (m2=8, m=4): `n_gam = C(8,4) = 70` gametes per genotype.
- **Ordered pairs** (original): n_gam² = 4,900 pairs per parent
- **Unordered pairs** (optimization): C(n_gam+1, 2) = 2,485 pairs — **49% fewer**

The key insight: offspring from {g₁ merged with g₂} equals offspring from {g₂ merged with g₁}. Heterogametic pairs (g₁ ≠ g₂) simply receive weight 2 instead of 1.

```r
het_pairs <- combn(n_gam, 2L)
idx1    <- c(seq_len(n_gam) - 1L, het_pairs[1L, ] - 1L)
idx2    <- c(seq_len(n_gam) - 1L, het_pairs[2L, ] - 1L)
pair_wt <- c(rep(1L, n_gam), rep(2L, ncol(het_pairs)))
```

Result: reduced gamete pair work by 49%, contributing to the overall runtime reduction.

## 7. Step 4 — Rcpp+OpenMP Selfmat + future Parallelism (Commit `4115954`, PR #3)

### Identifying the true bottleneck

Profiling the sparse path on the cb29/P8571 benchmark (17 alleles, 42,621 cone genotypes) showed that `.selfmat_cone_batch()` dominated runtime — building the sparse selfing matrix consumed the vast majority of the 152 s elapsed time.

### Analysis of possible optimizations

Three categories were evaluated:

**Remaining pure-R gains (~15–20% more):** `which()` + integer arithmetic, pre-computed index offsets — small marginal impact.

**Rcpp opportunities:**
- `.selfmat_cone_batch()` → **10–50× potential** — the dominant bottleneck
- `.build_cone()` → **20× potential** — second-largest pure-R loop
- `.merge_sorted_pairs()` → 3× — absorbed into the C++ selfmat

**Parallelism:** locus × population pairs are embarrassingly parallel — 8 loci × 2 pops = 16 fully independent computations, each requiring only read access to the shared input data.

### The C++ selfmat (`src/selfmat_cone.cpp`)

The algorithm in C++:
1. **Hash table**: `std::unordered_map<double, int>` mapping polynomial cone hashes to row indices — replaces R's `match()` for offspring lookup
2. **OpenMP parallel chunk loop**: each chunk of parent genotypes processed by one thread, with thread-local `std::vector<Triplet>` to avoid locking
3. **Inline merge-sort**: gamete alleles merged into an `int off[8]` stack array — zero heap allocation per pair
4. **Aggregate after parallel section**: sort + deduplicate triplets, write sparse (i, j, x) output

Key compile error encountered and resolved: `Rcpp::IntegerVector` does not support `.reserve()`. Fix — accumulate into `std::vector<int>` then wrap at return:
```cpp
std::vector<int> si_out, sj_out;
std::vector<double> sx_out;
// ... fill ...
return List::create(Named("i") = IntegerVector(si_out.begin(), si_out.end()), ...);
```

### The future locus×pop dispatch

`de_silva_freq_EM()` was refactored into a cleaner two-level structure:
1. Extract `.one_locus_pop()` internal helper (~100 lines of inner loop logic)
2. Build a task list: `expand.grid(L = loci, pop = pops)`
3. Serial path (`nthreads=1`): `lapply(tasks, ...)`
4. Parallel path (`nthreads>1`): `future::plan(multisession)` + `future_lapply()`

New parameter: `nthreads = 1L` — controls R-level worker processes. The number of C++ OpenMP threads (`cpp_threads`) is derived from `nthreads` (see Step 5 for the corrected formula).

### Challenge: Git Bash/MSYS2 incompatibility with PSOCK workers

`future::multisession` creates PSOCK socket workers — separate `Rscript.exe` processes. This crashed (exit code 139) from Git Bash/MSYS2 even for trivial `future_lapply(1:4, identity)`, due to a known Windows socket incompatibility in that shell environment. The code is correct; testing requires a native Windows terminal or RStudio.

### Result
- cb29/P8571: **152 s → 33.5 s (4.5× speedup)**, 361 MB → 247 MB (−32% memory)
- 169/173 tests pass — no regressions

## 8. Step 5 — Rcpp build_cone + Thread Partitioning Fix (Commit `0214a0b`)

### Gap 1: `.build_cone()` still in pure R

After the selfmat was ported to C++, `.build_cone()` became the next significant bottleneck. It contained three slow pure-R operations:

| Operation | Problem |
|---|---|
| Per-phenotype `GENLIST_r()` calls | 818 R function calls, each allocating an `ng × m2` matrix |
| `do.call(rbind, all_rows)` + `!duplicated()` | ~50K-row matrix creation |
| 42,621-iteration `indexf()` loop | 42K R function dispatches for INDEXF |

**`src/build_cone.cpp`** implements all three in a single C++ function:

1. **Inline GENLIST**: iterative multiset enumeration, directly mapping local→global allele indices via a `gmap[]` array — no intermediate matrix allocation
2. **Polynomial hash dedup**: `std::unordered_map<double, int>` with base `na1+1`, first-occurrence semantics
3. **C++ INDEXF**: direct translation of the `.make_indexf()` closure using exact `choose_int()` and `G_r()` inline functions — verified against the R formula for m=0,1,2,3
4. **Compact phenotype mapping**: `std::sort` + `std::unique` + `unordered_map`

The polysat `Genotype()` S4 accessor loop (which collects the observed phenotypes from the data) stays in R — it uses S4 method dispatch that cannot be called from C++.

**Result:** user time −41%, sys/GC time −79% on the benchmark pair.

### Gap 2: suboptimal thread partitioning

Original thread allocation strategy (incorrect):
```r
cpp_threads <- if (nthreads == 1L) parallel::detectCores() else 1L
```

With `nthreads=4` (4 R workers): each worker got `cpp_threads=1` — OpenMP was **completely disabled** in parallel mode, negating the selfmat speedup for parallel runs.

Fixed strategy:
```r
cpp_threads <- max(1L, parallel::detectCores() %/% nthreads)
```

With 12 cores and `nthreads=4`: each R worker gets 3 OpenMP threads. The total compute capacity stays the same (12 threads), properly distributed between R-level and C++-level parallelism.

## 9. Step 6 — Progress Reporting (Commit `2ea1db5`)

### The problem

The dense path produced **zero output** during computation. For a multi-minute full hornbeam run, the user had no feedback about which locus was running or how many iterations were needed. The sparse path had scattered messages in inconsistent formats, including intermediate noise lines ("Building sparse selfing matrix...", "Starting EM...") that added clutter without useful information.

### The solution

A unified format for every locus×population pair, covering both code paths:

```
ℹ [cb29 / P8571]  sparse path  (ng = 1,081,575)
ℹ   cone: 42,621 genotypes, 818 phenotypes       ← sparse path only
✔ [cb29 / P8571]  65 iterations
```
```
ℹ [loc1 / Pop1]   dense  path  (ng = 495)
✔ [loc1 / Pop1]   25 iterations
```

**Implementation details:**
- Start message emitted after `use_sparse` is determined, before any heavy computation
- Cone size line kept for the sparse path (useful diagnostic: see how much the cone reduces from full ng)
- Unified `cli_alert_success` completion line placed *after* the `if/else` block — a single code point covers both paths
- `quiet = TRUE` for parallel workers suppresses all output (interleaved output from multiple workers is unreadable)

## 10. Testing Strategy

### Pre-written test suite (`tests/testthat/`)

106 tests were written *before* implementation began, covering all 9 functions:

**`test-helpers.R`** (23 tests for 5 helper functions):
- `G_r`: correctness vs manual computation, edge cases (q=0, n=0)
- `INDEXG_r`: round-trip with GENLIST — `INDEXG(GENLIST[g,]) == g` for all g
- `GENLIST_r`: correct row count, sorted non-decreasing order, all values in 1..na1
- `RANMUL_r`: row sums of `arep = m2`, `rmul` positive, correct multinomial coefficients
- `SELFMAT_r`: column sums = `C(m2,m)²` (smatdiv), square matrix, non-negative entries

**`test-de_silva_freq_EM.R`** (17 tests for main function):
- Output structure matches polysat format exactly (column names, row names, `Genomes` column)
- Allele frequencies sum to 1 per locus per population
- Numerical agreement with `polysat::deSilvaFreq()` within tolerance 1e-8
- Edge cases: self=0 (pure random mating), self close to 1, genbinary input format
- Method parameter: "auto", "dense", "sparse" all produce consistent results

### Ad-hoc benchmark and verification scripts

**`test_sanity.R`**: runs both dense and sparse paths on the simulated `simgen` dataset, verifies:
- `max(abs(dense_result - sparse_result)) < 1e-8` — paths produce numerically identical results
- All frequency column sums = 1.000000 per population

**`test_subset.R`**: benchmark script for the cb29/P8571 locus×pop pair — the hardest single case (17 alleles, 42,621 cone genotypes). Measures elapsed time, user time, sys time, and peak memory via `proc.time()` and `gc()`. Ran after every optimization step to quantify the gain.

### Testing philosophy: fail fast with pre-written tests

Writing 106 tests *before* implementation had several advantages:
1. Defined exact expected behaviour before any code was written — no ambiguity about what "correct" means
2. Caught numerical errors immediately (especially off-by-one errors in INDEXF and INDEXG index formulas)
3. Provided confidence during refactoring — if all 169 tests passed after a refactoring, the refactoring was safe
4. Made the 4 pre-existing polysat failures (self=1 singularity, names mismatch) clearly identifiable from the start, without obscuring any regressions introduced by polyfreq

## 13. Lessons Learned: Claude-Assisted R Package Development

### What worked well

**1. Write tests first, implement second**
Pre-writing 106 tests before any implementation forced clarity about expected behaviour and caught bugs immediately. Claude could run the test suite after every implementation step and know instantly whether the code was correct.

**2. Start pure-R, profile, then selectively add C++**
The pure-R implementation was a prerequisite — it established correctness and gave something concrete to profile. Without a correct pure-R baseline, it would have been impossible to verify the C++ implementations. Jumping straight to C++ would have mixed algorithmic bugs with performance bugs and made debugging much harder.

**3. Separate mathematical complexity from performance complexity**
The de Silva algorithm involves non-trivial mathematics (Neumann series convergence, INDEXF combinatorial formula, genotype space enumeration). By working through all of this in R first — where the code is readable, interactive, and debuggable — mathematical correctness was established before any performance work began. The C++ ports were then straightforward mechanical translations of already-verified R logic.

**4. Iterative performance engineering with measured baselines**
Each optimization step was benchmarked against the previous baseline using `test_subset.R`. This made gains concrete and prevented regressing. The decision sequence was strictly data-driven:
- Baseline measurement → identify dominant bottleneck → port to C++ → re-measure
- Repeat until runtime target is met or marginal gains become negligible

**5. Reference implementation as ground truth**
Having `polysat::deSilvaFreq()` as a reference implementation was invaluable. Every refactoring step ended with:
```r
max(abs(polyfreq_result - polysat_result))  # must be < 1e-8
```
This single number told us immediately whether the refactoring was safe.

**6. Sparse matrix approach enabled the impossible**
The union phenotype cone was the single most impactful architectural decision: reducing the 1.08M genotype space to 42,621 (−96%) made octoploid analysis feasible at all. This required understanding the mathematical structure (cone closure under selfing, Neumann series convergence) before any code was written. The mathematical insight came from reading the original paper — the code was secondary.

### Challenges encountered

**Algorithm fidelity**: Several subtle numerical choices in the original C++ (`-744` for `log(0)`, exact convergence criterion, `smatdiv` scaling) had to be discovered by comparing outputs with the reference, not by reading documentation.

**Windows environment constraints**: `future::multisession` PSOCK workers crash from Git Bash/MSYS2. This blocked testing of the parallel R path throughout the project. The workaround: implement the code correctly, document the environment requirement, test the serial OpenMP path only from bash, and note that testing parallel mode requires native Windows terminal or RStudio.

**DLL locking on Windows**: R keeps the compiled `.dll` loaded across sessions, causing `devtools::install()` to fail with "cannot remove earlier installation" when a DLL is in use. Workaround: `R CMD INSTALL --no-lock --no-staged-install`.

**Rcpp API surprises**: `Rcpp::IntegerVector` does not support `.reserve()` — this caused a compile error that was not obvious from the Rcpp documentation. Fix: accumulate into `std::vector<int>` and wrap at the return statement.

**`make` caching**: After adding a new `.cpp` file, `make` may use cached `.o` files and skip recompilation ("Nothing to be done for 'all'"). Fix: delete all `.o` and `.dll` files before `R CMD INSTALL`.

**Thread over-subscription bug**: The initial thread partitioning logic (`cpp_threads = 1` when `nthreads > 1`) effectively disabled OpenMP in parallel mode. The bug was identified through reasoning about the code rather than benchmarking — it only becomes observable when actually running with `nthreads > 1`, which was blocked by the Git Bash/PSOCK issue.

### Workflow pattern for each optimization

For each new feature or optimization, the interaction with Claude followed a consistent cycle:

1. **Analyse** — identify the bottleneck (profiling, code reading, estimation)
2. **Plan** — design the implementation approach in plan mode; user reviews and approves before any code is written
3. **Implement** — write code file by file with explicit edits, explaining each change
4. **Verify** — `test_sanity.R` (correctness), `test_subset.R` (benchmark), `devtools::test()` (regression suite)
5. **Commit** — descriptive commit message capturing what changed, why, and the benchmark numbers
6. **Repeat**

This cycle kept each PR focused and the git history readable as a story of progressive, measured improvement. The plan-mode step was particularly valuable for larger changes (Rcpp implementation, parallel dispatch) where multiple architectural choices had to be made before writing any code.
